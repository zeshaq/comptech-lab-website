## Decision

Each operator batch must include the full lifecycle in one record:

| Phase | Required outcome |
| --- | --- |
| Preflight | Issue, milestone, phase, ADR coverage, cluster health, OADP posture, catalog/package/channel/CSV/image availability, and GitOps render evidence. |
| Install | Only batch-owned GitOps manifests are applied. Scope stays limited to the target cluster and operator family. |
| Validation | Argo health, Subscription, InstallPlan, CSV, operator pods, excluded operands, ClusterOperators, pods, CSRs, and domain checks are recorded. |
| Rollback or promotion | GitOps rollback path and CRD cleanup boundary are explicit. Forced CRD deletion requires a separate issue. |
| Documentation | The batch gets one coherent blog chapter and the operator version-lock reference is updated when package state changes. |

This preserves the user's requirement to avoid many separate installation
manual pages for every preflight, install, validation, and rollback step while
still making each completed operator batch visible in the manual index.

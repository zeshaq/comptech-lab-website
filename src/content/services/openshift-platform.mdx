We install, harden, and hand over the OpenShift platforms that run regulated workloads. Our
reference is a **disconnected v6 fleet** — a hub cluster plus spoke clusters, image-mirrored
from scratch, with internal DNS, internal registries, internal GitLab, internal CAs, and
Vault as the root of credential trust.

## What we deliver

- **Greenfield disconnected installation.** From bootstrap VM and `oc-mirror` through hub-cluster
  install, GitOps bootstrap, ACM/MCE, RHACS, and OADP. The full path is captured in a
  step-by-step manual you keep.
- **Fleet operating model.** Hub-and-spoke topology with ACM, pull-mode GitOps, Vault for
  secret material, ESO to bridge Kubernetes Secrets, RHACS for runtime security, OADP for
  backups, cert-manager + internal CA for trust.
- **Application delivery path.** Federated GitLab + dual-CI build paths, Quay for image
  custody, Nexus for artifacts, supply-chain signing, OpenShift Pipelines for runtime
  build/test/promote.
- **BFSI production hardening.** Production checklists derived from real bank engagements —
  air-gapped operation, dual-control admin access, RBAC posture, audit log capture, replay
  drills.
- **Handover.** Architecture decisions (ADRs), runbooks per failure mode, install manual,
  history-and-replay log. The post-engagement state is fully documented.

## Where this lives in our lab

Our reference fleet (`hub-dc-v6` + `spoke-dc-v6`) runs continuously. The same install path is
[documented end-to-end as the greenfield manual](/case-studies/greenfield-ocp-deployment/), and the
operating model is [the CompTech fleet case study](/case-studies/comptech-openshift-fleet/).

## Engagement shape

A typical platform engagement runs **8–14 weeks** and ends with a working fleet, a documented
install path, a runbook set, and an operating model your platform team owns. We don't take
operations long-term.

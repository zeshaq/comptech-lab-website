## The ask

BRAC Bank wanted a concrete answer to one question: **can a modern banking SPA — with real
payment integration and embedded customer chat — run cleanly on OpenShift, in a posture
that's recognisably production-shaped?**

A POC, but a POC with the same architectural moves you'd want in production: SPA-first
front-end, server-side chat backed by JBoss EAP, payment integration through the bank's
existing rails, OpenShift Pipelines for build, internal registry for images.

## What we delivered

- **SPA architecture.** A single-page front-end that maps cleanly to micro-frontend
  decomposition later. Static assets served from OpenShift Routes behind the edge.
- **JBoss EAP chat backend.** Customer-support chat as a JBoss EAP service, integrated with
  the SPA over WebSockets. JBoss chosen because it matched BRAC's existing skill base — a
  faster path to production than introducing a net-new runtime.
- **Bank payment integration.** A payment flow that exercises the bank's existing payment
  rails end-to-end, not a stub. Auth, retry semantics, and idempotency tested.
- **OpenShift Pipelines.** Build → scan → push to internal registry → deploy via GitOps.
- **Architecture writeup.** Every non-obvious decision captured as an ADR, with the trade-offs
  considered and the path forward for production.

## What it proved

- OpenShift is the right substrate for a regulated banking SPA, including the awkward bits
  (JBoss alongside modern containers, payment integration with strict idempotency).
- The platform-engineering moves we operate in our reference fleet — supply chain, secret
  custody, GitOps — translate cleanly to BFSI environments without invented exotica.

## Where this goes next

The POC is structured for a production hardening engagement: RHACS tuning for the BFSI risk
profile, identity through WSO2 IS or Ping, payment-flow audit-log capture, OADP backup of
chat history.
